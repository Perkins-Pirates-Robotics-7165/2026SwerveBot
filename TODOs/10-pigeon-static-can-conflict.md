# TODO: Fix PigeonSubsystem Static Field and Potential CAN ID Conflict

**Severity:** Medium  
**File:** `src/main/java/frc/robot/subsystems/PigeonSubsystem.java`

## Problem

`PigeonSubsystem` has two related issues:

### 1. Static Pigeon2 on a SubsystemBase

The `Pigeon2` and all methods are `static`. The subsystem is never instantiated in `RobotContainer`. This is architecturally confusing — a `SubsystemBase` with no instance and no command requirements provides no scheduler integration.

```java
private static final Pigeon2 pigeon = new Pigeon2(PigeonConstants.pigeonID);
```

The static field is initialized the first time the class is loaded, which could happen unpredictably.

### 2. Potential CAN ID Conflict with the Swerve Drivetrain

`PigeonConstants.pigeonID = 0`. The swerve drivetrain generated by Tuner X (`CommandSwerveDrivetrain` via `TunerConstants`) almost certainly manages its own `Pigeon2` internally for odometry. If both use the same CAN ID, they will both attempt to own the same device and produce conflicting reads.

Check `TunerConstants.java` or the tuner project to confirm whether the drivetrain's internal IMU is the same Pigeon2 at ID 0.

## Fix

**If the drivetrain already uses this Pigeon2:** Remove `PigeonSubsystem` entirely and read orientation from `CommandSwerveDrivetrain` directly (it exposes `getState().Pose` and odometry). This is the preferred approach — the drivetrain's odometry thread already fuses the IMU at high frequency.

**If a separate Pigeon2 is needed:** Change `PigeonSubsystem` to a normal instantiable class with instance fields and methods, instantiate it in `RobotContainer`, and resolve any CAN ID conflict.

## Steps

1. Open `TunerConstants.java` (auto-generated) and look for the `Pigeon2` CAN ID used by the drivetrain.
2. Compare it with `PigeonConstants.pigeonID = 0`.
3. If they conflict, determine if `PigeonSubsystem` is needed at all or if its usages can be replaced by drivetrain state data.
4. Either remove `PigeonSubsystem` or fix its architecture and CAN ID.
